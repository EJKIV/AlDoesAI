[4:45-6:00] ERROR HANDLING

[On screen: Error response JSON]

GraphQL errors look different from REST. You usually get a 200 status
code even when something fails. Check the errors array in the response.

The most common errors we hit:

TEXT_TOO_LONG - You're over the platform limit. Twitter is 280, others
are more generous. Buffer doesn't auto-truncate—you need to handle
this yourself.

PROFILE_NOT_FOUND - Your channel ID changed or got disconnected. Just
re-run the channels query and update your environment variables.

RATE_LIMITED - You're making too many requests. Buffer allows roughly 50
writes per minute. Implement exponential backoff:

[Show retry code - 10 seconds]

Always check error.extensions.code to handle specific cases.

[6:00-7:30] RATE LIMITING

Buffer returns rate limit info in response headers. Look for
X-RateLimit-Remaining and X-RateLimit-Reset.

In practice, stay well under the documented limits. If they say 50 per
minute, aim for 40. This gives you headroom for retries and handles
traffic spikes.

[Show rate limiter implementation - 10 seconds]

For batch operations, add delays between posts. Don't fire 20 requests
simultaneously—process them sequentially with a one or two second gap.
